# 크루 · 랜드마크 기획

> 상태: **Phase A 구현됨 / B·C 미구현** — 확정 개발 순서 A→B→C  
> 목적: Phase별로 구현할 때 이 문서를 기준으로 한다.  
> 최종 정리: 2026-07-24  
> API 출처: [한국관광공사_국문 관광정보 서비스_GW](https://www.data.go.kr/data/15101578/openapi.do)

---

## 1. 배경과 방향

### 현재 (구현됨)

- 개인이 깃발을 꽂고, 개인끼리 상대 깃발을 점령한다.
- 깃발 cost: 100 / 300 / 500 / 1000
- 반경: cost별 100 / 150 / 200 / 300m
- 점령 확률: 10 / 25 / 50 / 75%
- 전투력: 보유 active 깃발 cost 합
- 팀/크루 개념 없음

### 처음에 검토했다가 보류한 안

- 크루원이 같은 깃발에 포인트를 쌓아 키우고, 합동 점령(동참 알림·풀 모금·동시 재실)으로 뺏는 방식
- UX·실시간·어뷰징·지분 처리 부담이 커서 **1차로 채택하지 않음**

### 채택한 방향

1. **랜드마크(관광지 등)를 지도에 미리 둔다** — 쟁탈의 무대
2. 랜드마크 안에서는 **개인 깃발 꽂기·점령**만으로 싸운다 (합동 UX 없음)
3. **크루는 집계 단위** — 존 안 깃발 cost를 크루(또는 개인)별로 합산해 명칭 권리·Top10에 사용
### 확정 개발 순서

이 순서로만 진행한다. 앞 Phase가 동작한 뒤에 다음 Phase로 넘어간다.

| Phase | 한 줄 | 완료 기준 |
|-------|--------|-----------|
| **A** | TourAPI 연동 → 관리자 큐레이션 → **지도에 랜드마크 표시** | 관리자가 가져온 장소를 노출 켜면 유저 지도에 존(200m)이 보인다 |
| **B** | 랜드마크 영역 안 **개인 깃발 꽂기·점령** (+ 개인 합산·괄호·Top10) | 존 안에서 5m 깃발로 싸우고, 개인 기준으로 명칭/Top10이 돌아간다 |
| **C** | **크루** (집계 버킷·탭·하이라이트·괄호에 크루 표시) | 크루 합산이 개인과 섞여 1위·Top10에 반영된다 |

합동 점령 등 §5 비범위는 위 세 Phase 밖에서 재기획한다.

## 2. 랜드마크

### 정의

- 지도에 미리 표시되는 고정 구역 (한국관광공사 TourAPI 동기화 및/또는 관리자 등록)
- 기본 영향 반경: **약 200m**
- 기존 `premium_places`(상점·쿠폰)와 **별도 엔티티**

### 표시

- 지도 마커 + 200m 범위
- 1위 주체가 있으면 이름 옆에 괄호 표기 (지하철역 스폰서 표기 느낌)
  - 크루 1위: 크루 프로필 이미지 + 크루명
  - 개인 1위: 닉네임
- 선택 시 정보 영역: 기본 정보 + **합산 전투력 Top10** (개인·크루 혼합)

### 운영 상태 (폐역)

- 랜드마크는 **삭제하지 않는다**
- 의미가 없어지면 **폐역(비활성)** 처리
- 폐역 시:
  - 상세 설명에 **「현재 운영되지 않는 랜드마크」** 안내
  - **깃발 꽂기·점령·집계·명칭·Top10은 그대로 유지** (규칙 엔진을 끄지 않음)
- 삭제·반경 복귀 기획은 없음

### 데이터 출처

- 상세는 **§6 TourAPI 활용 설계** 참고
- 관리자: 후보 적재 → 지도 노출/폐역 → 반경 오버라이드·수동 추가

## 3. 랜드마크 안 깃발 규칙

| 항목 | 규칙 |
|------|------|
| cost | 기존과 동일 (100 / 300 / 500 / 1000) |
| 반경 | **항상 5m** (cost별 대형 반경 사용 안 함) |
| 개수 | 유저당 개수 제한 없음. **5m 충돌만** 준수하면 여러 개 가능 |
| 소속 판정 | 깃발 **중심 좌표**가 랜드마크 200m 안이면 해당 랜드마크 소속. **반경이 겹치면 모든 해당 랜드마크에 동시 적용** |
| 외부 깃발과의 충돌 | 랜드마크 **밖** 일반 깃발의 큰 반경이 존 **안** 배치를 막지 않음 |
| 존 안 신규 | 항상 랜드마크형(5m) |
| 점령 | 기존 개인 점령 규칙과 동일 (확률·비용·방어 보상) |

### 랜드마크 지도 노출 ON 시 기존 깃발 편입

- 관리자가 **지도 노출을 켤 때**, 중심이 존 반경 안인 기존 깃발은 **반경을 5m로 줄여 편입**하고 합산에 포함한다
- 편입 후 5m 기준으로 드물게 겹치더라도 **기존 깃발은 유지** (신규만 5m 규칙 적용)
- 폐역·노출 OFF로 반경을 다시 키우지 않음 → 복귀·겹침 폭발 문제 없음

### 현행 충돌 로직과의 관계

- 현재 `findActivePinsNear`는 기존·신규 반경 중 큰 값으로 충돌한다
- 랜드마크 존 안 배치는 **존 안 깃발끼리 5m 기준**으로만 충돌 검사하고, **존 밖 깃발 반경은 무시**해야 한다

## 4. 합산 · 명칭 권리 · Top10

### 집계 단위

- 대상: 해당 랜드마크 존 안 **active** 깃발
- 크루 소속 유저의 깃발 → **그 크루 버킷**에 cost 합산
- 미소속 → **개인 버킷** (닉네임으로 표시)
- Top10: **개인·크루 혼합** 주체 순위
- 반영: **실시간** (꽂기 / 점령 / 삭제 등 성공 직후)

### 명칭 (괄호) 1위

- 합산 전투력 1위 주체를 랜드마크명 옆에 표시
- **동점**이면 **해당 점수에 먼저 도달한 주체**
  - 정의: 그 주체의 합산이 현재 1위 점수에 **처음 도달한 시각**이 더 이른 쪽
  - 점수가 깨졌다가 다시 동점이 되면 **다시 도달한 시각** 기준

### 닉네임

- 개인 1위·Top10 개인 항목은 **닉네임** 표시
- 비속어는 **닉네임 설정 시** 차단 (괄호 표시 전제 과제)

### 크루 (랜드마크 단계에서의 역할)

- 이 단계의 크루는 “합동 점령 주체”가 아니라 **집계·표시 단위**
- 크루 생성/가입/프로필/하단 크루 탭 등은 **랜드마크·TourAPI 이후**에 구현
- 크루 미구현 기간에는 집계가 전부 개인 버킷이어도 규칙과 모순 없음

## 5. 크루 (후속 기획 요약)

TourAPI · 랜드마크 존·깃발·(가능하면) 개인 단위 합산/표시가 안정된 뒤 진행한다.

### 제품 스케치 (후속)

- 하단 **크루** 메뉴: 크루 목록 → 가입 → 소속 크루 현황 (인원, 멤버, 전투력, 크루 내 랭킹)
- 지도: **내 크루 깃발 하이라이트** 스위치
- 랜드마크 괄호·Top10에 크루 프로필·이름 반영
- 1인 1크루 등 운영 규칙, 가입/탈퇴/리더는 크루 구현 착수 시 상세 확정

### 명시적 비범위 (1차 크루에서도 하지 않기로 한 것)

- 크루원이 한 깃발에 포인트를 함께 쌓아 키우기
- 합동 점령 (개시·동참 알림·풀 모금·동시 재실)
- 크루 전용 깃발 소유로 개인 깃발을 대체하는 모델

## 6. TourAPI 활용 설계

### 6.1 API 개요

| 항목 | 내용 |
|------|------|
| 서비스 | 한국관광공사 국문 관광정보 (TourAPI 4.x / KorService2) |
| 포털 | https://www.data.go.kr/data/15101578/openapi.do |
| Base URL | `https://apis.data.go.kr/B551011/KorService2` |
| 규모 | 약 26만 건, 기능 15종 |
| 비용 | 무료 |
| 트래픽 | 개발계정 일 1,000회 수준 → **앱 지도에서 실시간 호출 금지**, DB 큐레이션 후 서빙 |
| 필수 공통 파라미터 | `serviceKey`, `MobileOS`, `MobileApp`, `_type=json` |
| 성공 | `response.header.resultCode == "0000"` |

사진(공공누리 1·3유형)은 CI/BI·명예훼손 용도 금지. 앱 내 장소 썸네일·상세 안내용으로만 사용하고 출처(한국관광공사)를 고지한다.

| `areaBasedSyncList2` | 대량 동기화는 운영계정·배치 여유 있을 때 |

### 6.4 앱에 저장할 필드 매핑

TourAPI 목록/상세 → 우리 `landmarks` (가칭)

| TourAPI | DB 컬럼 | 용도 |
|---------|---------|------|
| `contentid` | `tour_content_id` (UNIQUE) | 동기화 키, 재조회 |
| `contenttypeid` | `tour_content_type_id` | 필터·표시 |
| `title` | `name` | 지도·괄호 왼쪽 이름 |
| `mapy` | `lat` | 위도 (문자열→number) |
| `mapx` | `lng` | 경도 |
| `addr1` (+ `addr2`) | `address` | 상세 |
| `firstimage` | `image_url` | 마커/시트 썸네일 |
| `tel` | `tel` | 선택 표시 |
| `overview` (`detailCommon2`) | `overview` | 상세 설명 |
| `areacode` / `sigungucode` | `area_code` / `sigungu_code` | 관리자 필터 |
| `modifiedtime` | `source_modified_at` | 갱신 판단 |
| — | `radius_meters` DEFAULT 200 | 게임 존 (API에 없음) |
| — | `map_visible` | **지도 노출** (관리자) |
| — | `is_closed` | **폐역** (설명만, 규칙 유지) |
| — | `source` | `'tourapi'` \| `'manual'` |
| — | `admin_note` | 운영 메모 |

좌표 없는 항목·`mapx`/`mapy` 파싱 실패 항목은 **지도 노출 후보에서 제외**.

### 6.5 핵심 원칙: API ≠ 지도 직행

```
TourAPI (후보 소스)
    ↓ 관리자 검색 / 지역 가져오기
landmarks 테이블 (전체 카탈로그, 기본 map_visible=false)
    ↓ 관리자가 「지도 노출」 ON
유저 지도 API (map_visible=true 만)
```

- 유저 앱은 **우리 DB만** 읽는다. TourAPI 키·쿼터를 클라이언트에 노출하지 않는다.
- 전국 26만 건을 한 번에 넣지 않는다. **지역·키워드로 가져와 큐레이션**한다.
- `premium_places`와 테이블·메뉴를 분리한다 (광고/쿠폰 vs 쟁탈 존).

말씀하신 **「관리자에 API 데이터를 뿌리고 → 지도 노출 여부 결정」** 흐름과 일치한다.

### 6.6 관리자 「랜드마크 관리」 IA

프리미엄 장소 관리와 같은 패턴 (`AdminShell` 콘텐츠관리 메뉴 추가).

**메뉴:** `/admin/landmarks`

**목록 필터**

1. **지도 노출 중** — `map_visible=true`
2. **카탈로그** — DB에 있으나 미노출
3. **폐역** — `is_closed=true`
4. 검색: 이름, 지역코드, contentType, Tour contentId

**주요 액션**

| 액션 | 동작 |
|------|------|
| TourAPI에서 가져오기 | 지역(`areaBasedList2`) 또는 키워드/위치 검색 → 후보 upsert (`map_visible` 기본 false) |
| 지도 노출 ON | `map_visible=true`. **최초 ON 시** 존 안 기존 깃발 5m 편입(§3) — Phase B |
| 지도 노출 OFF | `map_visible=false` (마커 숨김). 이미 편입된 깃발 반경은 되돌리지 않음 |
| 폐역 | `is_closed=true` + 안내 문구. 꽂기/점령/집계는 유지(§2) |
| 수동 추가 | `source=manual`, 지도에서 좌표 지정 (`AdminLocationMap` 재사용) |
| 반경 수정 | `radius_meters` 오버라이드 (기본 200) |
| 상세 새로고침 | `detailCommon2`로 overview/주소/이미지 갱신 |

**노출 OFF vs 폐역**

| | 지도 노출 OFF | 폐역 |
|--|---------------|------|
| 맵 마커 | 숨김 | `map_visible`이면 **표시** + 상세에 폐역 안내 |
| 게임 규칙 | (마커 없으니 사실상 신규 유입 없음) | 꽂기·점령·집계 **유지** |
| 용도 | 아직/더 이상 무대가 아님 | “운영 종료” 카피만 |

1차: 폐역이어도 `map_visible`이면 맵에 남긴다. 폐역=자동 숨김은 하지 않는다.

### 6.7 유저 지도 (Phase A 범위)

**Phase A에서 할 것**

- [ ] `landmarks` 스키마 + 마이그레이션
- [ ] 서버 TourAPI 클라이언트 (`TOUR_API_SERVICE_KEY`는 서버 전용)
- [ ] 관리자 랜드마크 메뉴 (가져오기 / 노출 / 폐역 / 수동 추가)
- [ ] 유저 `GET` 랜드마크 목록 (bbox 또는 근처, `map_visible=true`)
- [ ] 지도 마커 + 200m 원 + 탭 시 이름·주소·개요·이미지
- [ ] 폐역 시 상세 안내 문구
- [ ] 출처 고지 (한국관광공사)

**Phase A에서 하지 않는 것**

- 존 안 5m 깃발·충돌 예외·편입 (Phase B)
- 합산·괄호 명칭·Top10 (Phase B)
- 크루 (Phase C)

괄호 자리(`남산타워 (○○크루)`)는 UI 여백만 두거나, Phase B 전까지는 이름만 표시.

### 6.8 크루 시스템과의 연계 (데이터·화면 계약)

크루는 나중에 붙이되, **지금 만드는 랜드마크 모델이 계약**이 된다.

```
landmark (고정 무대, TourAPI/수동)
  └── pins whose center ∈ radius   ← Phase B
        └── owner → crew_id?       ← Phase C
              └── SUM(cost) by crew|user → title holder + Top10
```

| 이후 기능 | 지금 준비할 것 | 나중에 붙일 것 |
|-----------|----------------|----------------|
| 괄호 1위 | `landmarks.name`, 안정적 `id` | 실시간 집계 또는 holder 캐시 필드 |
| Top10 시트 | 랜드마크 상세 바텀시트 골격 | `/api/landmarks/[id]/ranking` |
| 크루 이미지 | — | 크루 프로필을 1위 배지에 |
| 지도 하이라이트 | 랜드마크 레이어 ≠ 깃발 레이어 | “내 크루 깃발” 토글 |
| 편입 | `map_visible` false→true 전환 훅 자리 | 존 안 pin radius 일괄 5m |

**관리자 「지도 노출 ON」이 게임 무대 오픈 스위치**다.  
크루 없이도 개인 합산만으로 명칭 전쟁이 가능하고, 크루는 같은 집계에 `crew` 버킷만 추가하면 된다.

### 6.9 환경·운영 메모

- `.env.local`: `TOUR_API_SERVICE_KEY` (data.go.kr 발급 키). `NEXT_PUBLIC_` 금지.
- `MobileApp`: 서비스명(`이땅내땅` 등) 고정.
- 개발 트래픽 한도: 관리자 “가져오기”만 TourAPI 호출, 이후는 DB.
- 외부 이미지 도메인: `next.config` 허용 여부 확인.

## 7. 확정 개발 순서 (상세)

§1 「확정 개발 순서」와 동일. 여기서는 작업 단위만 풀어 적는다.

### Phase A — 랜드마크를 지도에 뿌리기 (다음 착수)

1. `landmarks` 스키마·타입
2. TourAPI 서버 클라이언트 + 관리자 가져오기/노출/폐역/수동
3. 유저 지도 마커·200m·상세 시트·출처 고지  
→ **크루·존 깃발 규칙 없음.** 표시와 운영 스위치만.

### Phase B — 존 안 개인 정복

1. 존 안 깃발 5m · 외부 반경이 존 배치를 막지 않음 · 다중 배치
2. 지도 노출 ON 시 기존 깃발 5m 편입
3. 기존과 동일한 개인 점령 루프
4. 개인 버킷 합산 · 동점 선도달 · 괄호·Top10 · 닉네임 비속어 필터  
→ **크루 없이** “랜드마크 명칭 전쟁”이 성립해야 함.

### Phase C — 크루

1. 크루 생성/가입/프로필·하단 탭
2. 집계에 크루 버킷 추가 · 괄호/Top10 크루 표시
3. 지도 내 크루 깃발 하이라이트  
→ Phase B 파이프에 크루만 얹음. 합동 점령은 비범위.
